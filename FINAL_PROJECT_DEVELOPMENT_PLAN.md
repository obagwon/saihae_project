# 사이해 기말 프로젝트 발전 기획서

## 1. 프로젝트 개요

**사이해**는 “사이를 이해하다”와 “나를 이해하다”라는 의미를 가진 Flutter 기반 자기이해 앱이다. 현재 구현은 감정 체크, 오늘의 멘탈 팁, 성향 테스트, 성향별 관계 가이드, 감정 기록 저장으로 구성되어 있으며, 전문 심리 진단이 아니라 일상에서 사용자가 자신의 감정과 관계 성향을 가볍게 돌아보는 참고용 서비스라는 방향이 코드와 README 전반에 반영되어 있다.

기말 프로젝트로 발전시키기 위한 최종 목표는 기능 수를 무리하게 늘리는 것이 아니라, **감정 기록 → 추천 → 성향 이해 → 기록/통계 확인**이 하나의 반복 가능한 사용자 흐름으로 연결되는 완성형 모바일 앱을 만드는 것이다.

## 2. 저장소 분석 결과

### 2.1 사용 기술과 프레임워크

- Flutter/Dart 프로젝트이다. `pubspec.yaml`의 `environment.sdk`는 `^3.11.1`이며, 앱 진입점은 `lib/main.dart`의 `runApp(const SaihaeApp())`이다.
- UI는 Material 3 기반이며 `lib/app/theme.dart`의 `AppTheme.lightTheme`에서 파스텔 컬러, 텍스트 스타일, 버튼, 카드, 하단 내비게이션 테마를 정의한다.
- 로컬 저장은 `shared_preferences` 패키지를 사용한다. `lib/services/local_storage_service.dart`의 `LocalStorageService`가 감정 기록을 JSON 문자열로 직렬화해 저장한다.
- 외부 패키지는 런타임 기준 `shared_preferences`, `cupertino_icons`가 있으며, 개발 도구로 `flutter_lints`, `flutter_launcher_icons`가 설정되어 있다.

### 2.2 전체 폴더 및 파일 구조

```text

│  ├─ emotion_data.dart       # 감정 가이드 하드코딩 데이터
│  ├─ personality_data.dart   # 성향 유형 하드코딩 데이터
│  └─ tip_data.dart           # 오늘의 멘탈 팁 하드코딩 데이터
├─ models/
│  ├─ emotion_record.dart     # 감정 기록 모델
│  ├─ mental_tip.dart         # 멘탈 팁 모델

├─ screens/
│  ├─ analysis_screen.dart    # 나 분석 탭
│  ├─ home_screen.dart        # 홈 탭
│  ├─ main_tab_screen.dart    # 4개 탭 하단 내비게이션
│  ├─ record_screen.dart      # 감정 기록 탭
│  ├─ relation_guide_screen.dart # 관계 가이드 탭
│  ├─ result_screen.dart      # 성향 테스트 결과 화면
│  └─ test_screen.dart        # 성향 테스트 진행 화면
├─ services/
│  └─ local_storage_service.dart # SharedPreferences 저장 서비스
└─ widgets/

   ├─ section_title.dart
```

루트에는 `README.md`, `pubspec.yaml`, `analysis_options.yaml`, 플랫폼별 Flutter 생성 폴더(`android`, `ios`, `web`, `macos`, `linux`, `windows`), 앱 아이콘(`assets/icons/app_icon.png`), README용 실행 화면 이미지(`images/*.jpg`)가 있다.

### 2.3 앱 실행 흐름

1. `lib/main.dart`에서 `SaihaeApp` 실행.
2. `SaihaeApp`은 `MaterialApp`을 만들고 `home`으로 `MainTabScreen`을 지정.
3. `MainTabScreen`은 4개 탭을 보유한다: `HomeScreen`, `AnalysisScreen`, `RelationGuideScreen`, `RecordScreen`.
4. 홈 또는 나 분석 탭에서 `Navigator.push`로 `TestScreen` 진입.
5. `TestScreen`에서 8개 질문에 답하면 `calculateResult()`로 `PersonalityType`을 계산하고 `Navigator.pushReplacement`로 `ResultScreen` 표시.
6. 기록 탭에서는 감정과 메모를 입력해 `LocalStorageService.saveEmotionRecord()`로 저장하고, `getEmotionRecords()`로 최근 기록을 다시 불러온다.

### 2.4 구현된 화면 목록

| 화면 | 파일 | 클래스 | 구현 상태 | 역할 |
|---|---|---|---|---|
| 앱 루트 | `lib/app/app.dart` | `SaihaeApp` | 구현 | 테마와 첫 화면 설정 |
| 메인 탭 | `lib/screens/main_tab_screen.dart` | `MainTabScreen` | 구현 | 4개 탭 전환 |
| 홈 | `lib/screens/home_screen.dart` | `HomeScreen` | 구현 | 오늘의 감정 체크, 멘탈 팁, 테스트 진입 |
| 나 분석 | `lib/screens/analysis_screen.dart` | `AnalysisScreen` | 구현 | 성향 테스트 소개 및 결과 예시 |
| 성향 테스트 | `lib/screens/test_screen.dart` | `TestScreen` | 구현 | 질문 진행 및 결과 계산 |
| 결과 | `lib/screens/result_screen.dart` | `ResultScreen` | 구현 | 성향 결과와 관계 힌트 표시 |
| 관계 가이드 | `lib/screens/relation_guide_screen.dart` | `RelationGuideScreen` | 구현 | 성향 유형별 관계 팁 목록 |
| 기록 | `lib/screens/record_screen.dart` | `RecordScreen` | 구현 | 감정+한 줄 회고 저장/조회/삭제 |

### 2.5 화면 간 이동 구조

- 기본 이동은 `MainTabScreen`의 `BottomNavigationBar`로 처리한다.
- `HomeScreen`과 `AnalysisScreen`은 `TestScreen`으로 `Navigator.push`한다.
- `TestScreen` 완료 시 `ResultScreen`으로 `Navigator.pushReplacement`한다.
- `ResultScreen`의 “다시 테스트하기”는 `TestScreen`으로 `pushReplacement`, “홈으로 돌아가기”는 `Navigator.popUntil(context, (route) => route.isFirst)`를 사용한다.
- 기록 화면은 별도 상세 화면 없이 같은 탭에서 입력, 최근 목록, 전체 삭제를 처리한다.

### 2.6 현재 구현된 주요 기능

- 감정 선택: `emotionGuides`의 6개 감정을 `EmotionChip`으로 선택한다.
- 감정별 메시지/팁: `EmotionGuide.message`, `EmotionGuide.tip`를 홈과 기록 화면에서 표시한다.
- 오늘의 멘탈 팁: `DateTime.now().day % mentalTips.length`로 날짜별 팁을 선택한다.
- 성향 테스트: `TestScreen.questions`의 8개 질문과 4단계 응답으로 결과를 계산한다.
- 결과 카드: `ResultCard`가 성향명, 설명, 강점, 스트레스 반응, 잘 맞는 사람, 추천 행동을 표시한다.
- 관계 가이드: `personalityTypes` 전체를 순회해 유형별 관계 팁을 보여준다.
- 감정 기록 저장: `EmotionRecord`를 JSON으로 변환해 `shared_preferences`에 저장한다.
- 최근 기록 조회: 저장된 기록을 앱 내 목록으로 표시한다.
- 전체 삭제: `clearEmotionRecords()`로 기록 전체를 삭제한다.

### 2.7 데이터 저장 방식

- 저장 대상은 감정 기록뿐이다.
- `EmotionRecord`는 `emotion`, `memo`, `createdAt` 필드를 가진다.
- `LocalStorageService`는 `_emotionRecordsKey = 'emotion_records'`로 `SharedPreferences`에 JSON 문자열을 저장한다.
- 성향 검사 결과, 사용자 닉네임, 온보딩 완료 여부, 앱 설정, 즐겨찾기, 통계 캐시 등은 저장하지 않는다.

### 2.8 상태 관리 방식

- 별도의 상태관리 패키지는 없다.
- `HomeScreen`, `TestScreen`, `RecordScreen`, `MainTabScreen`에서 `StatefulWidget`과 `setState`를 사용한다.
- 앱 규모가 작아 현재는 적절하지만, 기록/통계/검사 결과 저장이 추가되면 서비스와 상태가 화면에 분산될 가능성이 있다.

### 2.9 모델, 서비스, 위젯, 화면의 책임 분리 상태

- 모델: `EmotionRecord`, `MentalTip`, `PersonalityType`은 단순 데이터 구조만 담당한다.
- 데이터: `emotion_data.dart`, `tip_data.dart`, `personality_data.dart`가 샘플/하드코딩 콘텐츠를 담당한다.
- 서비스: `LocalStorageService`는 감정 기록 저장만 담당한다.
- 위젯: `SoftCard`, `RoundedButton`, `EmotionChip`, `ResultCard`, `SectionTitle`로 공통 UI가 분리되어 있다.
- 화면: 일부 비즈니스 로직이 화면에 남아 있다. 특히 `TestScreen` 내부에 `TestQuestion` 모델, 질문 데이터, 점수 계산 로직이 함께 있어 재사용성과 테스트 가능성이 낮다.

### 2.10 외부 패키지와 라이브러리

| 패키지 | 위치 | 용도 | 평가 |
|---|---|---|---|
| `shared_preferences` | `pubspec.yaml` | 로컬 감정 기록 저장 | 현재 규모에 적합 |
| `cupertino_icons` | `pubspec.yaml` | iOS 스타일 아이콘 제공 | 직접 사용 여부 확인 필요 |
| `flutter_lints` | `pubspec.yaml` | 정적 분석 규칙 | 필요 |
| `flutter_launcher_icons` | `pubspec.yaml` | 앱 아이콘 생성 | README와 설정에 부합 |

### 2.11 assets 구성

- `assets/icons/app_icon.png`가 앱 아이콘 원본으로 존재하고 `flutter_launcher_icons.image_path`로 설정되어 있다.
- `images/*.jpg`는 README 스크린샷 용도이며, `pubspec.yaml`의 Flutter assets에는 등록되어 있지 않다.
- 앱 내부에서 이미지 asset을 직접 사용하는 코드는 확인되지 않았다.

### 2.12 미완성 또는 임시 구현 기능

- 홈의 감정 체크는 선택 후 메시지만 보여주며 저장과 연결되지 않는다.
- 기록은 “오늘의 감정”을 저장하지만 감정 강도, 날짜별 조회, 통계, 중복 저장 제어는 없다.
- 성향 테스트 결과는 화면에만 표시되고 저장되지 않는다.
- 나 분석 탭의 결과 예시는 `personalityTypes.first`를 사용하므로 실제 사용자 결과와 무관하다.
- 오늘의 팁은 날짜의 일(day)에 따른 단순 순환이다.
- 테스트 코드가 Flutter 기본 카운터 테스트 형태로 남아 현재 앱과 맞지 않는다.

### 2.13 코드에 존재하지만 실제로 사용되지 않거나 효과가 제한적인 기능

- `test/widget_test.dart`는 `MyApp`을 참조하지만 실제 `main.dart`에는 `MyApp`이 없고 `SaihaeApp`만 존재한다. 따라서 테스트는 실패할 가능성이 높다.
- README의 “Android 미니 프로젝트” 설명과 달리 저장소에는 iOS, web, macOS, linux, windows 플랫폼 폴더도 포함되어 있다. 다만 실제 타깃은 Android 중심으로 보인다.
- `cupertino_icons`는 의존성에 있으나 현재 확인한 주요 Dart 파일에서는 직접 import되지 않는다.

### 2.14 하드코딩된 데이터

- 감정 목록과 안내 문구: `lib/data/emotion_data.dart`의 `emotionGuides`.
- 멘탈 팁: `lib/data/tip_data.dart`의 `mentalTips`.
- 성향 유형과 결과 콘텐츠: `lib/data/personality_data.dart`의 `personalityTypes`.
- 성향 검사 질문과 점수표: `lib/screens/test_screen.dart`의 `questions`.
- 탭 목록, 타이틀, 문구 대부분도 화면 파일에 직접 작성되어 있다.

### 2.15 오류가 발생할 가능성이 있는 부분

- `LocalStorageService.getEmotionRecords()`는 JSON 파싱 실패, 필드 누락, 타입 불일치에 대한 예외 처리가 없다.
- `EmotionRecord.fromJson()`은 `DateTime.parse` 실패 가능성을 처리하지 않는다.
- `RecordScreen.loadRecords()`는 `setState` 전 `mounted` 체크가 없다. 일반 사용에서는 큰 문제가 드물지만 비동기 로딩 중 화면이 dispose되면 위험하다.
- 성향 테스트 동점 처리 기준이 명시되어 있지 않고, `Map.entries.first`에서 시작해 더 큰 점수만 교체하므로 동점이면 앞에 선언된 유형이 선택된다.
- `flutter analyze`는 현재 환경에 Flutter SDK가 없어 실행하지 못했다. 다만 테스트 파일은 코드상 `MyApp` 미정의 문제로 실패가 예상된다.

### 2.16 UI와 기능이 연결되지 않은 부분

- 홈 감정 체크와 기록 저장이 연결되지 않는다. 사용자는 홈에서 감정을 선택해도 기록 탭에 자동 반영되지 않는다.
- 성향 결과와 관계 가이드가 데이터는 공유하지만, 사용자의 실제 결과를 관계 탭에서 우선 표시하거나 저장하지 않는다.
- 추천 콘텐츠는 감정/성향별 텍스트 팁 수준이며 음악/영화/책/행동 추천 모델은 없다.
- 기록 목록은 최근 기록만 보여주며 날짜별 조회나 통계 화면과 연결되지 않는다.

### 2.17 유지보수성과 확장성을 방해하는 구조

- `TestQuestion`과 질문 데이터가 화면 파일 내부에 있어 테스트 데이터 수정과 UI 수정이 결합되어 있다.
- 저장 서비스가 감정 기록만 지원해 성향 결과, 사용자 설정, 온보딩 상태 저장을 확장하려면 서비스 구조를 넓혀야 한다.
- 감정 값이 `emoji + name` 문자열로 저장되어 통계 계산 시 감정 ID 기반 집계가 어렵다.
- 저장 기록에 감정 강도, 날짜 키, 추천 ID, 사용자 메모 길이 등의 구조화 필드가 없다.
- 모든 화면이 `setState` 기반이라 기능 증가 시 상태 흐름 추적이 어려워질 수 있다.

### 2.18 README와 실제 구현 내용의 차이

- README의 주요 기능 설명은 대부분 실제 구현과 일치한다.
- README는 “감정 기록 및 한 줄 회고 저장”을 설명하고 실제 `RecordScreen`과 `LocalStorageService`가 이를 구현한다.
- README는 “날짜 기준으로 멘탈 팁이 바뀜”이라고 설명하며 실제로 `DateTime.now().day % mentalTips.length` 방식이다.
- README의 실행 방법은 `flutter pub get`, `flutter run`이지만 현재 분석 환경에는 `flutter` 명령이 없어 실행 확인은 하지 못했다.
- README의 라이선스 항목은 MIT License라고 쓰여 있으나 저장소 루트에서 `LICENSE` 파일은 확인되지 않았다. 제출 시 실제 라이선스 파일을 추가하거나 README 문구를 조정하는 것이 좋다.

## 3. 현재 구현 기능

1. **오늘의 감정 체크**: 6개 감정 중 하나를 선택하면 감정 안내와 팁을 표시한다.
2. **오늘의 작은 회복 팁**: 날짜 기반으로 하나의 멘탈 팁을 표시한다.
3. **성향 테스트**: 8개 질문, 4단계 응답, 5개 성향 유형 중 하나 산출.
4. **성향 결과 카드**: 성향 설명, 강점, 스트레스 반응, 잘 맞는 사람, 추천 행동 표시.
5. **관계 가이드**: 5개 성향 유형별 관계 팁, 친해지는 방법, 조심할 점 표시.
6. **감정 기록 저장**: 감정과 한 줄 회고를 로컬에 저장.
7. **최근 기록 조회/전체 삭제**: 저장 기록 리스트 표시 및 전체 삭제.
8. **공통 UI 컴포넌트**: 카드, 버튼, 감정 칩, 섹션 제목, 결과 카드.

## 4. 현재 프로젝트의 장점

- 콘셉트가 명확하다. “전문 진단이 아닌 참고용 자기이해”라는 문구가 홈, 테스트, 결과, README에 반복적으로 나타난다.
- 파스텔 색상, 둥근 카드, 부드러운 여백 등 감성 라이프스타일 앱의 방향이 UI 코드에 반영되어 있다.
- 기능이 작지만 홈, 분석, 관계, 기록의 탭 구성이 명확해 발표 시 설명하기 쉽다.
- `models`, `data`, `screens`, `widgets`, `services`로 기본 책임 분리가 되어 있다.
- 서버 없이도 실행 가능한 로컬 앱이라 기말 시연 안정성이 높다.
- 성향 데이터와 관계 가이드가 연결되어 있어 앱의 정체성을 보여주기 좋다.

## 5. 문제점 및 개선 필요 사항

### 5.1 기능 완성도 문제

- 홈에서 선택한 감정이 기록으로 이어지지 않아 핵심 사용자 흐름이 끊긴다.
- 기록은 단순 목록이며 날짜별 조회, 감정 통계, 주간 변화 등 반복 사용 동기가 약하다.
- 성향 테스트 결과가 저장되지 않아 앱 재실행 후 “나의 성향”을 유지하지 못한다.
- 성향별 추천은 결과 카드의 `dailyAction` 수준이라 “콘텐츠 추천 앱”처럼 보이기에는 부족하다.

### 5.2 데이터 구조 문제

- 감정 기록에 `emotionId`가 없고 문자열만 저장되어 통계 확장이 불편하다.
- 감정 강도, 작성 날짜의 일 단위 키, 수정 여부, 추천 콘텐츠 ID가 없다.
- 성향 검사 질문이 화면 내부에 있어 모델/데이터 분리가 부족하다.

### 5.3 안정성 문제

- 저장 데이터 손상 시 JSON 파싱 예외가 발생할 수 있다.
- 테스트 파일이 현재 앱 구조와 맞지 않는다.
- Flutter 실행 환경 확인이 필요하다.

### 5.4 발표 완성도 문제

- 현재 상태는 “예쁜 설문+기록 앱” 수준으로 보일 수 있다.
- 사용자가 반복적으로 사용할 이유를 통계/연속 기록/추천 저장 등으로 강화해야 한다.
- 빈 상태, 최초 실행 안내, 프로필/닉네임이 없어 완성형 서비스 느낌이 약하다.

## 6. 기말 프로젝트 목표

기말 제출 목표는 다음으로 설정한다.

> “사이해는 사용자가 오늘의 감정을 기록하고, 감정과 성향에 맞는 작은 추천을 확인하며, 누적된 기록을 통해 자신을 가볍게 돌아볼 수 있는 로컬 기반 자기이해 앱이다.”

핵심 완성 기준은 다음과 같다.

- 홈 감정 체크와 기록 저장의 흐름 연결.
- 감정 강도와 이유 메모 저장.
- 날짜별/최근 감정 기록 조회.
- 주간 감정 요약 또는 간단 통계 제공.
- 성향 테스트 결과 저장 및 재확인.
- 전문 진단이 아니라는 고지 명확화.
- 현재 기본 테스트를 실제 앱 테스트로 교체.

## 7. 핵심 사용자

| 사용자 | 필요 | 앱에서 제공할 가치 |
|---|---|---|
| 대학생 | 바쁜 일상 속 내 감정 정리 | 1분 감정 체크와 한 줄 회고 |
| 과제/시험 스트레스가 있는 사용자 | 복잡한 감정의 가벼운 정리 | 감정별 작은 행동 추천 |
| 관계 고민이 있는 사용자 | 나와 타인의 관계 스타일 이해 | 성향 테스트와 관계 가이드 |
| 꾸준한 기록이 어려운 사용자 | 부담 없는 기록 방식 | 짧은 선택형 입력과 최근 기록 확인 |

## 8. 핵심 사용자 시나리오

1. 사용자는 앱을 처음 실행하고 “전문 진단이 아닌 자기이해 참고용 앱”이라는 안내를 확인한다.
2. 홈에서 오늘의 감정을 선택하고 감정 강도를 고른다.
3. 감정을 선택한 이유를 한 줄로 작성한다.
4. 저장 후 오늘의 작은 추천 행동을 확인한다.
5. 기록 탭에서 오늘 기록이 추가된 것을 확인한다.
6. 나 분석 탭에서 성향 테스트를 진행한다.
7. 결과 화면에서 나의 성향과 관계 팁, 추천 행동을 확인한다.
8. 이후 기록/통계 화면에서 최근 7일 감정 변화와 자주 나타난 감정을 확인한다.

## 10. 최종 추천 기능 구성

### 버전 1: 안정적인 완성도 중심

- 포함 기능: 온보딩, 감정 기록 저장 개선, 감정 강도, 날짜별 최근 기록, 성향 결과 저장, 실제 widget smoke test.
- 제외 기능: 그래프, 공유 이미지, 알림, 로그인, AI.
- 핵심 흐름: 온보딩 → 홈 감정 체크 → 기록 저장 → 최근 기록 확인 → 성향 테스트 → 결과 저장.
- 구현 난이도: 낮음~중간.
- 장점: 실패 가능성이 낮고 현재 코드와 잘 맞는다.
- 단점: 시각적 차별성이 약할 수 있다.
- 예상 위험: 저장 구조 변경 시 기존 기록 호환 처리 필요.
- 추천 대상: 시간이 3일 이하인 경우.
- 현재 프로젝트 적합도: 높음.

### 버전 2: 기능과 차별성의 균형 — 최종 추천

- 포함 기능: 버전 1 전체 + 자주 나타난 감정 통계 + 최근 7일 감정 강도 미니 그래프 + 성향/감정 기반 추천 카드.
- 제외 기능: 로그인, 클라우드, AI, 알림, 결과 이미지 생성.
- 핵심 흐름: 온보딩 → 오늘 감정/강도/이유 기록 → 즉시 추천 확인 → 기록/통계 확인 → 성향 테스트 → 성향 기반 추천 확인.
- 구현 난이도: 중간.
- 장점: 반복 사용 이유와 시연 가치가 모두 생긴다.
- 단점: 통계 UI와 데이터 모델 개선 작업이 필요하다.
- 예상 위험: 차트 구현을 과하게 잡으면 일정 초과 가능성이 있다. 외부 차트 패키지 대신 단순 막대 UI를 권장한다.
- 추천 대상: 1주일 이상 개발 가능하고 기말 완성도를 높이고 싶은 경우.
- 현재 프로젝트 적합도: 매우 높음.

### 버전 3: 높은 도전 수준

- 포함 기능: 버전 2 전체 + 즐겨찾기 + 결과 공유 이미지 + 다크 모드 + 연속 기록.
- 제외 기능: 로그인, 서버, AI 챗봇.
- 핵심 흐름: 기록/추천/공유/즐겨찾기까지 앱 서비스 느낌 강화.
- 구현 난이도: 높음.
- 장점: 발표 시 기능이 풍부해 보인다.
- 단점: 핵심 흐름보다 부가기능이 많아질 수 있다.
- 예상 위험: 일정 초과, UI 일관성 저하, 테스트 부족.
- 추천 대상: 2주 이상 안정적으로 개발 가능한 경우.
- 현재 프로젝트 적합도: 중간.

**최종 추천은 버전 2**이다. 현재 프로젝트는 이미 감정, 성향, 기록의 뼈대가 있으므로 통계와 추천을 얹으면 “반복 사용 가능한 자기이해 앱”으로 보이기 가장 좋다. 서버나 AI 없이도 완성도와 시연 효과를 확보할 수 있다.

## 12. 데이터 구조

### 12.1 권장 모델

```dart
class EmotionRecord {
  final String id;
  final String emotionId;
  final String emotionLabel;
  final String emoji;
  final int intensity; // 1~5
  final String memo;
  final DateTime createdAt;
}

class EmotionType {
  final String id;
  final String label;
  final String emoji;
  final String message;
  final String tip;
}

class PersonalityTestResult {
  final String typeId;
  final DateTime testedAt;
  final Map<String, int> scores;
}

class RecommendationItem {
  final String id;
  final String category; // action, book, music 등. 기말 범위는 action 권장
  final List<String> emotionIds;
  final List<String> personalityTypeIds;
  final String title;
  final String description;
}

class AppSettings {
  final bool onboardingCompleted;
  final String? nickname;
  final bool disclaimerAccepted;
}
```

### 12.2 모델 간 관계

- `EmotionRecord.emotionId`는 `EmotionType.id`와 연결된다.
- `PersonalityTestResult.typeId`는 `PersonalityType.id`와 연결된다.
- `RecommendationItem`은 감정 ID 또는 성향 ID 목록과 매칭된다.
- 통계는 `EmotionRecord` 목록에서 계산한다.

### 12.3 로컬 저장 방식

- 기말 범위에서는 `shared_preferences` 유지 권장.
- 키 분리 권장:
  - `emotion_records_v2`
  - `personality_result`
  - `app_settings`
  - `favorite_recommendations`는 시간이 남을 때만.
- 기존 `emotion_records`는 마이그레이션하거나 개발 중이면 초기화해도 된다. 제출 직전에는 샘플 데이터 준비 기능을 별도로 두는 것이 안전하다.

### 12.4 상태 관리 방식

- 3일 일정: 현재처럼 `setState` 유지.
- 1주일 일정: `LocalStorageService`를 확장하고 화면별 로딩 함수만 정리.
- 2주일 일정: `ChangeNotifier` 기반 `AppState` 또는 `ValueNotifier`를 도입해 기록/성향 결과를 여러 화면에서 공유.
- Riverpod/Bloc은 현재 규모와 기말 일정에서는 과도하다.

### 12.5 추천 로직

- 1차: 오늘 감정에 맞는 행동 추천.
- 2차: 저장된 성향 결과가 있으면 성향별 문장을 추가.
- 3차: 감정 강도가 4~5이면 회복 중심 추천, 1~2이면 유지/확장 중심 추천.
- 위기 감정 안내는 전문 상담처럼 표현하지 말고 “강한 불안/무기력함이 오래 지속되면 주변 사람이나 전문 기관 도움을 고려해보세요” 수준으로 표시한다.

### 12.6 통계 계산 방식

- 자주 나타난 감정: `records.groupBy(emotionId).length`.
- 최근 7일 강도: 날짜별 평균 `intensity`.
- 기록 횟수: 전체 길이와 이번 주 길이.
- 연속 기록: 날짜 set을 만들고 오늘부터 역순으로 연속 날짜 계산.
- 기록이 없으면 “아직 통계가 없어요. 3일만 기록하면 작은 패턴을 볼 수 있어요.” 같은 빈 상태를 표시한다.

### 12.7 샘플 데이터와 실제 사용자 데이터 구분

- 발표용 샘플 데이터가 필요하면 `debugSeedRecords()` 같은 개발용 함수는 제출 전 숨기거나 README에 시연 준비 방법으로만 남긴다.
- 실제 저장 데이터와 샘플 콘텐츠는 키 또는 플래그로 구분한다.
- 앱 첫 실행에서 자동으로 가짜 기록을 넣는 것은 사용자 신뢰를 낮출 수 있으므로 피한다.

### 12.8 리팩터링 필요 부분

- `TestQuestion`을 `lib/models/test_question.dart`로 이동.
- 질문 목록을 `lib/data/test_question_data.dart`로 이동.
- 결과 계산을 `lib/services/personality_test_service.dart`로 분리.
- 감정 데이터를 `EmotionGuide`에서 `EmotionType` 모델로 정리.
- `LocalStorageService`에 감정 기록 외 성향 결과/설정 저장 메서드 추가.

### 12.9 기말 프로젝트 수준에서 불필요하게 복잡한 기술

- Firebase Auth/Firestore.
- 자체 백엔드 서버.
- AI API 기반 감정 분석/챗봇.
- sqflite 기반 관계형 DB.
- 복잡한 차트 패키지와 고급 애니메이션.
- 푸시 알림.

## 14. 구현 우선순위

### A. 기말 제출 전 반드시 구현할 핵심 기능

#### A1. 감정 기록 모델 확장

- 사용자에게 필요한 이유: 단순 메모가 아니라 감정 강도와 날짜 패턴을 볼 수 있어 반복 사용 이유가 생긴다.
- 현재 코드 연결 지점: `EmotionRecord`, `LocalStorageService`, `RecordScreen`.
- 수정 가능 파일: `lib/models/emotion_record.dart`, `lib/services/local_storage_service.dart`, `lib/screens/record_screen.dart`, `lib/data/emotion_data.dart`.
- 구현 방식: `emotionId`, `emoji`, `label`, `intensity` 필드 추가. 기존 JSON 파싱은 기본값을 넣어 호환.
- 난이도: 중간.
- 예상 작업량: 3~5시간.
- 우선순위: 1.
- 시연 장점: “오늘은 불안 4점, 이유는 시험”처럼 구체적 기록을 보여줄 수 있다.
- 문제: 기존 저장 데이터와 충돌 가능성.

#### A2. 홈 감정 체크와 기록 흐름 연결

- 이유: 홈에서 감정만 선택하고 끝나는 단절을 해결한다.
- 연결 지점: `HomeScreen`, `RecordScreen`, `MainTabScreen`.
- 파일: `home_screen.dart`, `main_tab_screen.dart`, 필요 시 `emotion_record_form.dart` 위젯.
- 구현 방식: 홈 카드에 “이 감정으로 기록하기” 버튼 추가, 선택 감정을 기록 화면 초기값으로 전달하거나 기록 폼을 공통 위젯화.
- 난이도: 중간.
- 작업량: 3~4시간.
- 우선순위: 2.
- 시연 장점: 사용자가 자연스럽게 기록까지 이어진다.
- 문제: 탭 간 초기값 전달 구조 설계 필요.

#### A3. 성향 결과 저장 및 나 분석 탭 표시

- 이유: 테스트 결과가 사라지지 않아 앱이 개인화된 느낌을 준다.
- 연결 지점: `TestScreen.calculateResult()`, `ResultScreen`, `AnalysisScreen`, `LocalStorageService`.
- 파일: `personality_test_result.dart`, `local_storage_service.dart`, `analysis_screen.dart`, `result_screen.dart`.
- 구현 방식: 결과 typeId와 날짜, 점수표 저장. 분석 탭에서 저장된 결과가 있으면 미리보기 대신 실제 결과 표시.
- 난이도: 중간.
- 작업량: 4~6시간.
- 우선순위: 3.
- 시연 장점: “내 성향 기반 추천”으로 서비스 완성도 상승.
- 문제: `calculateResult()`가 점수표를 반환하도록 구조 개선 필요.

#### A4. 기록/통계 섹션 추가

- 이유: 반복 사용 가치를 만든다.
- 연결 지점: `RecordScreen.records`, `EmotionRecord.createdAt`.
- 파일: `record_screen.dart`, `emotion_stats_service.dart`, 필요 시 `stat_card.dart`.
- 구현 방식: 자주 나타난 감정, 최근 7일 평균 강도, 기록 횟수를 카드로 표시.
- 난이도: 중간.
- 작업량: 5~8시간.
- 우선순위: 4.
- 시연 장점: 데이터 처리 능력을 평가자에게 보여준다.
- 문제: 기록이 적으면 통계가 빈약하므로 시연 데이터 준비 필요.

#### A5. 온보딩과 비진단 고지 강화

- 이유: 심리 진단 오해를 줄이고 앱 신뢰도를 높인다.
- 연결 지점: `SaihaeApp`, `LocalStorageService`, README 문구.
- 파일: `app.dart`, `onboarding_screen.dart`, `app_settings.dart`, `local_storage_service.dart`.
- 구현 방식: 첫 실행 시 온보딩 표시 후 완료 여부 저장.
- 난이도: 낮음~중간.
- 작업량: 3~4시간.
- 우선순위: 5.
- 시연 장점: 기획 의도와 윤리성을 명확히 보여준다.
- 문제: 앱 시작 비동기 로딩 처리 필요.

#### A6. 테스트 코드 정상화

- 이유: 현재 기본 카운터 테스트는 앱과 맞지 않아 평가상 감점 요인이다.
- 연결 지점: `test/widget_test.dart`, `SaihaeApp`.
- 파일: `test/widget_test.dart`.
- 구현 방식: `SaihaeApp`을 pump하고 `사이해`, `홈`, `나 분석`, `기록` 같은 텍스트가 보이는지 확인.
- 난이도: 낮음.
- 작업량: 1~2시간.
- 우선순위: 6.
- 시연 장점: 기본 품질 관리 강조 가능.
- 문제: Flutter SDK 환경 필요.

### B. 시간이 남으면 추가할 차별화 기능

- 연속 기록 횟수: 기록 동기 부여에 좋다.
- 성향/감정 기반 추천 콘텐츠 확장: 행동 추천 중심으로 20~30개 정도 구성.
- 결과 공유 이미지: 시연 효과는 크지만 후순위.
- 즐겨찾기: 추천 콘텐츠가 충분할 때만 가치가 있다.
- 다크 모드: UI 완성도는 높지만 핵심 흐름 완료 후 진행.
- 접근성 개선: Semantic label, 대비, 터치 영역, 텍스트 크기 점검.

### C. 장기적으로 확장할 기능

- 로컬 DB 전환: 기록이 많아지고 검색/필터가 필요할 때.
- Firebase/클라우드 동기화: 여러 기기 사용과 백업이 필요할 때.
- 기관 제휴용 익명 통계: 실제 서비스화 시.
- 전문가 콘텐츠 큐레이션: 심리 진단으로 오해되지 않는 범위에서.
- AI 감정 분석: 안전 정책과 고지, 위기 대응 체계가 갖춰진 뒤.

## 17. 시연 시나리오

### 3~5분 시연 순서

1. **홈 화면 시작**: 앱 이름과 “자기이해 참고용” 콘셉트를 소개한다.
2. **오늘의 감정 체크**: “불안해요” 또는 “지쳤어요”를 선택하고 안내 문구가 바뀌는 것을 보여준다.
3. **감정 기록 저장**: 강도 4, 이유 “기말 발표 준비 때문에 마음이 복잡했다”를 저장한다.
4. **오늘의 추천 확인**: 감정에 맞는 작은 행동 추천을 보여준다.
5. **기록/통계 화면**: 미리 준비한 5~7개 기록으로 자주 나타난 감정과 최근 7일 강도 변화를 보여준다.
6. **성향 테스트 진행**: 2~3문항만 설명하고 빠르게 답변해 결과 화면으로 이동한다.
7. **결과 화면**: 성향명, 강점, 스트레스 반응, 관계 팁을 보여준다.
8. **나 분석 탭 재확인**: 저장된 성향 결과가 분석 탭에 남아 있음을 보여준다.
9. **마무리**: 서버 없이 로컬 저장으로 개인정보 부담을 낮췄고, 진단이 아닌 자기이해 참고용이라고 강조한다.

### 미리 준비할 데이터

- 최근 7일 감정 기록 5~7개.
- 서로 다른 감정 3종 이상.
- 강도 1~5가 섞인 데이터.
- 성향 테스트 결과 1개.

### 오류나 빈 화면 방지

- 시연 전 앱 데이터 초기화 여부를 확인한다.
- 기록이 없을 때도 빈 상태 문구가 자연스럽게 보이도록 한다.
- 통계 화면은 기록 3개 미만일 때 안내 문구를 보여준다.
- 네트워크가 없어도 모든 기능이 동작하도록 한다.

### 발표자가 강조할 기술적 포인트

- Flutter Material 3 기반 커스텀 테마.
- `shared_preferences` 기반 로컬 저장.
- JSON 직렬화/역직렬화.
- 성향 테스트 점수 계산 로직.
- 기록 데이터 기반 통계 계산.
- 모델/데이터/서비스/위젯/화면 분리.

### 발표자가 강조할 기획적 포인트

- 전문 심리 진단이 아니라 부담 없는 자기이해 앱.
- 감정 기록과 성향 이해를 하나의 흐름으로 연결.

  A. 기말 프로젝트 범위에서는 개인정보 부담을 줄이고 오프라인 시연 안정성을 높이기 위해 로컬 저장을 선택했습니다.

## 20. 최종 결론

현재 사이해 프로젝트는 이미 기획 방향, 기본 UI, 성향 콘텐츠, 감정 기록 저장이 갖춰져 있어 기말 프로젝트의 기반으로 충분하다. 다만 현재 상태 그대로는 홈 감정 체크, 기록, 성향 결과, 관계 가이드가 각각 분리되어 있어 “완성된 서비스”보다는 “여러 화면을 가진 과제 앱”으로 보일 가능성이 있다.

따라서 제출 전에는 **감정 기록 모델 확장, 홈→기록 연결, 성향 결과 저장, 기록 통계, 테스트 정상화**에 집중해야 한다. 서버, 로그인, AI, 알림 같은 고난도 기능은 현재 범위에서는 제외하고, 네트워크 없이 안정적으로 시연 가능한 로컬 앱으로 완성하는 것이 가장 합리적이다.

## 가장 먼저 구현해야 할 단 하나의 기능

**가장 먼저 구현해야 할 기능은 “감정 기록 모델 확장과 감정 강도 저장”이다.**

이유는 이 기능이 이후의 거의 모든 개선의 기반이기 때문이다. 감정 강도와 구조화된 감정 ID가 있어야 날짜별 조회, 자주 나타난 감정 통계, 최근 7일 그래프, 감정 기반 추천, 연속 기록 같은 기능을 안정적으로 만들 수 있다. 현재 기록은 `emoji + name` 문자열과 메모만 저장하므로 확장성이 낮다. 따라서 기말 제출 완성도를 높이려면 먼저 기록 데이터를 제대로 설계해야 한다.
